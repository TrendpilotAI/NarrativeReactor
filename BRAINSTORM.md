## 2. Code Quality

### DRY Violations
- **Publisher duplication**: `publisher.ts` has platform formatters (Twitter, LinkedIn) AND `blotatoPublisher.ts` wraps Blotato. Both have publishing logic. Consolidate into a unified `PublisherService` with adapters. **Effort: S**
- **Auth patterns**: API key auth is spread across middleware + inline in multiple places. Centralize.
- **Error handling**: Each service has its own try/catch pattern with different logging approaches. Extract a shared `ServiceError` base class.
- **AI client initialization**: `genkit.config.js` + multiple direct API client instantiations across services. Single `AIClientFactory`.

### Type Safety Gaps
- Several service return types are `any` or implicit
- `flows/orchestration.ts` likely has loose typing for agentic chat
- Missing Zod validation on webhook payloads

### Dead Code Suspects
- `postiz-app/` directory in root — likely scaffolding that was never integrated
- `debug-exports.js` in root — debugging artifact
- `scripts/test-fal-registry.ts` and `scripts/test-generate-content.ts` — ad-hoc test scripts that should be proper test cases

## 6. Performance

### Issues Found
- **No caching**: Every AI call hits the API fresh. Repeated brand voice/guidelines loads (`loadBrandGuidelines`, `loadStoryBibleContext`) should be cached in Redis.
- **Synchronous file reads**: `context.ts` likely reads files synchronously for brand context — should async/cache.
- **No queue**: Content generation jobs run inline in HTTP request handlers — long-running tasks should be queued (Bull/BullMQ on Redis).
- **No streaming**: Content generation responses could stream back to the client instead of waiting for full completion.
- **Cost tracking**: `costTracker.ts` uses in-memory tracking — resets on restart. Should persist to Redis/Postgres.

### Recommended
- BullMQ job queue for async content generation
- Redis caching for brand context, AI responses (TTL-based)
- Streaming SSE responses for long-running generation
- Persistent cost tracking in Postgres

## 7. Security

### Issues
- **CORS is `*`** (`origin: '*'` in index.ts) — should restrict to known origins in production
- **Webhook secret comparison** is plain string equality — should use `crypto.timingSafeEqual` to prevent timing attacks
- **No request body size limit** — Express default is 100kb but no explicit limit set; large payloads possible
- **API key stored in memory** — `apiKeyAuth` middleware likely reads from env but no rotation mechanism
- **No rate limiting on webhooks** — only `/api` routes are rate limited; webhooks endpoint is not
- **Dependency audit needed** — `genkit: "*"` wildcard versions could pull in vulnerable packages

### Recommended
- Fix CORS to specific origins
- Use `crypto.timingSafeEqual` for webhook secret
- Add rate limiting to webhook routes
- Lock down wildcard dependency versions
- Add `helmet` middleware for security headers
- Set up automated `npm audit` in CI
